# Personal workflow (Justin)

Shell workflow commands installed by dotfiles in the devcontainer (zsh functions from `~/dotfiles/cp/` and `~/dotfiles/clickup/`).  Full background/rationale for anything terse below lives in `~/dotfiles/.claude/workflow-reference.md` - read it on demand, it is deliberately NOT loaded every turn.

## Environment

- `ISSUE_BRANCH_PREFIX=justin`
- `GITHUB_DEFAULT_PR_REVIEWER=Carepatron/platform`
- Branch: `justin/CU-{taskid}-{slug}`.  PR title: `[CU-{taskid}] {Capitalized title}`.  Both are generated by the workflow commands - never hand-type them.

## Tool use (token efficiency)

- Prefer the native `Grep` / `Glob` / `Read` tools for searching and reading files.  Use `Bash` only for commands with no native equivalent (git, gh, aws, dotnet, tests, docker).  Native search/read is more token-efficient and bypasses the rtk output-rewrite hook, which otherwise truncates or mangles `grep` / `find` / diff output.
- Operate inside a worktree using **absolute paths**; do not rely on `cd` persistence - the Bash tool resets cwd to the project dir every call, so `cd $WT` churn is wasted and error-prone.

## Devcontainer worktree workflow

Trigger: "use devcontainer worktree workflow" / "worktree this" / any close variant -> use `--worktree` on `new` (new task) or `start` (existing task id).  Each task gets its own worktree under `~/worktrees/` so parallel Claude sessions don't collide on `/workspace`.

```zsh
new --worktree "Task title" [description]   # new task: ClickUp task + worktree + branch + symlinks + IN PROGRESS + sprint
start --worktree <task-id>                  # existing task id
```

`--worktree` routes through `git_worktree_for_task_branch` (in `~/dotfiles/cp/git.zsh`): fetches origin, `worktree add ~/worktrees/CU-{id}-{slug} -b {branch} origin/master`, and symlinks the standard node_modules paths.  Fails fast on conflict.  (Why the symlinks are needed + the shared-node_modules concurrency rule: see workflow-reference.md.)

After the worktree exists:

1. `cd ~/worktrees/CU-{id}-{slug}/` and work inside it with absolute paths.  Red/green TDD per `public-api/design/development.md`.
2. Commit in logical units - signed, hooks must run (see Git commit rules).
3. Open the PR: for public-api work use `/public-api-pr` (see PR creation).
4. Update the ExecPlan if one applies (tick checkboxes, add Surprises/Decisions).
5. Leave the worktree until the PR merges; post-merge run `cleanup`.  Don't manually `git worktree remove` / `branch -D` without approval.

Constraints:

- Never check out a feature branch in `/workspace` itself - it blocks parallel sessions.  All feature work happens in worktrees.
- Never run `yarn install` (or anything that mutates node_modules) from inside a worktree - all worktrees share one physical tree, so concurrent installs corrupt it.  Do installs in `/workspace`.  (Narrow exception in workflow-reference.md.)
- Never force-push, `reset --hard`, or `branch -D` without explicit approval.
- If a future master adds a node_modules root, add it to `worktree_paths` in git.zsh (discover: `find /workspace -maxdepth 4 -type d -name node_modules -not -path '*/node_modules/*'`).

## Implementing a plan (`implement @<...>.plan.md`)

"implement @<path>.plan.md" (or a close variant) is the standing public-api kickoff.  Defaults:

1. Use the worktree workflow (`start --worktree <task-id>` / `new --worktree`).  Exception: if told to use the current branch / `/workspace`, honour that.
2. Implement per the plan + repo standards: red/green TDD with visible cycles, signed commits that run hooks, keep the plan's Progress/checkboxes current.
3. On completion, PR via `/public-api-pr` (default review route).  Skip route ("no review" / "quick pr" / `--skip-review`) only if asked.  No `--ai-review`/greptile unless explicitly asked.

## PR creation - public-api uses `/public-api-pr`, never `/pr-create`

Any "make a PR" request on public-api work goes through `/public-api-pr`.  It wraps the `pr` shell function (ClickUp IN REVIEW, title/body, reviewer, browser-open) and adds the fresh no-context `/public-api-code-review` + CI loop.  Quick PR with no self-review: `/public-api-pr` with skip intent.  Outside public-api (e.g. ui/): fall back to the `pr` alias (`cp_pr_task`), still never `/pr-create`.  The bare `pr` alias is Justin's own manual route.  (Why never `/pr-create`: see workflow-reference.md.)

### `--ai-review` / `-ar` / `--greptile` - explicit opt-in only

Never pass `--ai-review` (or `-ar` / `--greptile`) unless the operator's current-request message explicitly asks for it (e.g. "pr with greptile", "use ai review").  Scope descriptions ("this is a big change") and vague "review carefully" do NOT count - ask if ambiguous.  Default is no flag for every PR.  (Full rationale in workflow-reference.md.)

## Git commit rules

- **Always signed.**  Never `--no-gpg-sign` / `-c commit.gpgsign=false` / alter signing, even defensively.  Signing is pre-configured (SSH key at `~/.ssh/id_ed25519_signing`) - trust it, don't probe it.  If a commit fails *because of* signing, surface the error and stop; don't retry with signing off.
- **Always run pre-commit hooks (lefthook).**  Never `--no-verify` / `-n` on commit/merge/rebase/cherry-pick/revert; never set `LEFTHOOK*` skip envs or lefthook skip flags.  If a hook fails, surface the full output and stop - fix the cause and re-commit.  If it fails for an unrelated reason (broken codegen, esbuild mismatch), stop and tell the operator.
- **Commit subjects: no `[CU-...]` prefix.**  Write a plain conventional subject (e.g. `HostEnvironment: introduce Backend`).  The `[CU-{taskid}]` prefix belongs only to PR titles and branch names, both auto-generated; it appears in `git log` only because squash-merges inherit the PR title.

All commit rules apply to every commit - feature, fixup, amend, rebase, cherry-pick - even on private branches, throwaway worktrees, or pre-squash.  (Full rationale for each in workflow-reference.md.)

## Workflow commands (summary)

Full flag reference in workflow-reference.md.

| Command | Does |
|---|---|
| `new` / `cp_new_task <title> [desc] [--no-assignment] [--no-start] [--worktree]` | Create ClickUp task, (by default) start it |
| `start` / `cp_start_task <task-id> [--worktree]` | Fetch task, create/checkout branch, IN PROGRESS, add to sprint.  Accepts id or ClickUp URL |
| `pr` / `cp_pr_task [--body D] [--ai-review] [--no-slack] [--channel C]` | Push, create/update draft PR (reviewer Carepatron/platform), IN REVIEW, Slack notify.  Prefer `/public-api-pr` |
| `cleanup` / `cp_cleanup_branches` | Remove worktrees/branches whose upstream is gone; mark tasks DONE |
| `clickup <cmd>` | ClickUp API: whoami / get-task / start-task / pr-task / complete-task / create-task / add-task-to-current-sprint |

## Investigating CI failures

Devcontainer `gh` uses a fine-grained PAT; only the status-check rollup is broken (`checks:read` ungrantable), so avoid `gh pr checks`.  Use `gh run list --branch <branch> --json databaseId,headSha,status,conclusion,workflowName`, then `gh api repos/<owner>/<repo>/actions/runs/<run-id>/jobs` to find the failing job, then `gh api .../actions/jobs/<job-id>/logs`.  `gh run view <run-id> [--log]` works (ignore its trailing ANNOTATIONS 403).  (Full scriptable path in workflow-reference.md.)
